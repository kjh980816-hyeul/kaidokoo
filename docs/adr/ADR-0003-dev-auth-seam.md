# ADR-0003 · 인증 자리(seam) = `X-Member-Id` 헤더 임시 신원 (소셜 로그인 도입 전)

- 상태: 확정 (2026-06-07) · 한시적(소셜 로그인 구현 시 폐기/교체)
- 맥락: P2 이후 커뮤니티/관리자 기능(댓글·좋아요·출석·등급·게시판/회원 관리·라이브 토글)을 구현하려면
  "현재 회원"과 "ADMIN 권한"이 필요하다. 그러나 이 서비스의 로그인은 **소셜 OAuth 3종(ADR-0001)** 뿐이고,
  소셜 앱 등록은 외부 키(Client-Id/Secret) 주입이 선행되어야 해 아직 착수 전이다.
  로그인 없이 나머지 기능을 먼저 완성하기 위한 신원 해석 방법이 필요하다.

## 결정
1. **임시 신원 출처 = `X-Member-Id` 요청 헤더.** `@CurrentMember` 파라미터를
   `CurrentMemberArgumentResolver`가 이 헤더의 회원 id로 해석한다.
2. **권한·상태는 항상 DB 값으로 서버 재검증.** 헤더는 "누구인지"만 가리키고,
   `role == ADMIN`(`AdminGuard`)·`status == ACTIVE` 검증은 DB에서 조회한 회원으로 수행한다.
   즉 클라이언트가 보낸 role을 신뢰하지 않는다(30-security 준수).
3. **교체 지점 단일화.** 신원 해석은 리졸버 한 곳에만 존재한다. 세션/JWT가 들어오면
   이 리졸버 내부만 세션 기반으로 바꾸면 되고, 컨트롤러·서비스의 `@CurrentMember`/`AdminGuard`는 불변.
4. **새 엔드포인트는 본문에 `authorId`를 받지 않는다.** 댓글/좋아요 등은 `@CurrentMember`로 작성자를 도출.
   (P1의 글 작성 `authorId` 본문 필드는 레거시로 남겨두고 인증 도입 시 함께 정리한다.)

## 근거
- 외부 키 의존(소셜 로그인)을 분리해, 키 없이도 도메인/관리자 기능 전체를 구현·테스트·시연할 수 있다.
- 권한 검증 로직을 미리 올바른 위치(서버·DB 기준)에 박아두므로, 인증 전환 시 보안 로직 재작성이 없다.

## 리스크 / 완화
- **헤더 위조 가능 = 운영 보안 아님.** 누구나 `X-Member-Id: <admin>`로 관리자 행세 가능.
  - 완화: 이는 **인증 도입 전 개발용 발판**임을 코드/주석/이 ADR에 명시. 운영 배포 전 소셜 로그인 + 세션을
    반드시 선행한다(미구현 상태로 공개 배포 금지). 프론트는 dev 전용 회원 전환 UI로만 헤더를 세팅.
- 신규 회원 기본 등급 자동 배정 등 "가입 플로우" 로직은 소셜 로그인과 함께 구현 → 지금은 시드/관리자 수동 배정으로 대체.
