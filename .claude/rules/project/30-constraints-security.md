# 30 · 보안 & 제약사항

## 시크릿 / 자격증명
- 씨미 `Client-Secret`, 토큰, DB 비밀번호, 세션 서명 키는 **환경변수/시크릿 매니저**에서만. 코드·로그·git·프론트 번들·이 문서에 평문 노출 금지.
- `.gitignore`에 `.env*`, 키 파일, 빌드 산출물 명시. 커밋 전 시크릿 스캔.

## 인증 / 세션
- 우리 세션 토큰은 서버에서 발급·검증. 만료/갱신/로그아웃(취소) 처리.
- 모든 보호 API는 서버에서 인증·권한 재검증. 클라이언트 신뢰 금지.
- OAuth 콜백의 `state` 검증 필수(CSRF). 리디렉트 URI 화이트리스트.

## 입력 / 출력
- 모든 사용자 입력 서버 검증(길이·타입·금칙어). 게시글 본문은 XSS 방지(허용 태그 화이트리스트 또는 이스케이프).
- 업로드 이미지: 확장자/MIME/용량 검증, 실행 파일 차단, 원본·썸네일 분리, 안전한 경로.
- SQL은 파라미터 바인딩(문자열 연결 금지). JPA/MyBatis 동적 쿼리도 바인딩.

## 권한 / 데이터
- 관리자 API는 `role == ADMIN` 재검증 + `admin_log` 기록.
- 신고/제재/숨김은 소프트 처리 + 이력 보존.
- 개인정보 최소 수집·최소 보관. 탈퇴 시 처리 정책 정의.

## 외부 의존 (씨미)
- 씨미 장애·만료에 대한 폴백 필수: 로그인 실패 시 명확한 안내, 라이브 폴링 실패 시 배너 OFF/이전 캐시 표시. 외부 장애가 사이트 전체를 막지 않게.
- 토큰 만료 시 자동 갱신 후 재시도. 무한 재시도 금지(백오프).

## 변경 통제
- DB는 마이그레이션으로만 변경. `ddl-auto=validate` 고정. 운영 수동 변경 금지.
- 비자명한 결정은 ADR로 박제. 룰과 충돌하는 구현은 룰을 먼저 고치고 진행(임의 위반 금지).

## 금지 (HARD)
- 시크릿 출력, 권한 우회, 클라이언트 측 보안 의존, 미검증 입력 신뢰, 운영 DB 수동 변경, 게시판/등급/채널ID 하드코딩.
